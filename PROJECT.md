# 프로젝트 브리프: 웹 앱 쇼케이스 사이트

## 1. 목적
- 내가 만든 웹 앱/서비스를 한곳에서 소개하는 개인 브랜드 허브
- 방문자가 각 앱을 바로 체험(라이브 데모)하도록 유도
- 장기적으로 수익화(모델 미정)와 홍보 기반이 되는 사이트
- 타깃: 잠재 사용자, 그리고 협업/채용/제휴 가능성이 있는 방문자

## 2. 페이지 구조
- `/` 홈: 한 줄 소개 + 대표 프로젝트 2~3개 하이라이트 + 전체 보기 링크
- `/projects` 목록: 카드 그리드 (썸네일, 이름, 한 줄 설명, 상태 뱃지, 태그 표시)
- `/projects/[slug]` 상세: 문제 → 해결 → 스크린샷/데모 → 기술 스택 → CTA(버튼 텍스트·링크는 frontmatter의 ctaText·liveUrl로 제어)
- `/about`: 소개, 연락처, GitHub/SNS 링크

## 3. 기술 요구사항
- 스택: Next.js (App Router) + Tailwind CSS + MDX
- 프로젝트 데이터: `content/projects/*.mdx` 파일 기반
  - frontmatter: `title`, `description`, `thumbnail`, `liveUrl`, `repoUrl`(선택), `ctaText`(선택, 기본 "사용해보기"), `status`(live | beta | archived), `tags`, `date`, `featured`(boolean)
  - 파일 추가만으로 목록/상세 페이지 자동 생성
- 배포: Cloudflare Workers 정적 자산 — 정적 내보내기(`output: 'export'`), 설정은 `wrangler.jsonc`(자산 디렉토리 `./out`), 빌드 커맨드 `npm run build`
  - 주소: 일단 `*.workers.dev` 서브도메인, 커스텀 도메인은 수익화 후 검토
- SEO: 페이지별 메타태그, OG 이미지(현 단계: 썸네일 재사용 — 자동 생성이 필요해지면 satori 빌드 스크립트로), sitemap.xml, robots.txt
- 분석: Cloudflare Web Analytics (무료)
- 반응형 필수, Lighthouse 성능/접근성 90+ 목표

## 6. 작업 원칙 (Claude Code용)
- 새 프로젝트 추가는 MDX 파일 1개 추가로 끝나야 함 — 코드 수정 불필요
- 컴포넌트는 재사용 가능하게, 과도한 추상화는 피할 것
- CTA 버튼 텍스트/링크는 frontmatter에서 제어 (수익화 모델 확정 시 교체 용이하게)
- 커밋은 기능 단위로 작게

## 7. 백로그 (미구현 기획 항목)
- 태그 필터 UI (`/projects`) — 착수 조건: 등록 프로젝트가 늘어나 목록 탐색이 불편해지면
- 이메일 구독 폼 연동 (푸터 + 프로젝트 상세 하단, Buttondown/ConvertKit 무료 티어) — 착수 조건: 수익화 방향 정해지면
- `/log` 빌드 로그/블로그 — 착수 조건: 공유할 빌드 로그/글이 쌓이기 시작하면
